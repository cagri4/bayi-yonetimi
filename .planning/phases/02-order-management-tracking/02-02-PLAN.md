Create server actions file with:

1. `getAdminOrders(filters)` function:
   - Accept filters: { page?: number, status?: string, dealerId?: string, from?: string, to?: string }
   - Verify user is admin (check users.role = 'admin')
   - Build Supabase query with .select() including dealer:dealers(id, company_name, email), status:order_statuses(id, code, name)
   - Apply filters conditionally:
     - .eq('status_id', status) if provided
     - .eq('dealer_id', dealerId) if provided
     - .gte('created_at', from) if provided
     - .lte('created_at', to) if provided
   - Use pagination: pageSize = 50, calculate offset
   - Order by created_at DESC
   - Return { orders, count, error }
   Reference 02-RESEARCH.md Pattern 3.

2. `updateOrderStatus(orderId, newStatusId, notes?)` function:
   - Verify admin role
   - Call validate_order_status_transition RPC function to check if transition is valid
   - If invalid, return { error: 'Gecersiz durum degisikligi' }
   - Update orders table with new status_id
   - Revalidate paths: /admin/orders, /admin/orders/[orderId]
   - Return { success: true } or { error: string }
   Note: Status history auto-created by database trigger from Plan 01.
   Reference 02-RESEARCH.md Code Examples - Server Action for Status Update.

3. `cancelOrder(orderId, reason?)` function:
   - Verify admin role
   - Get current order status
   - Only allow cancellation from 'pending' or 'confirmed' statuses (validate via order_status_transitions)
   - Get 'cancelled' status id
   - Call updateOrderStatus with cancelled status
   - Add reason to notes if provided
   - Return { success: true } or { error: string }

4. `getValidNextStatuses(orderId)` function:
   - Get current status_id from order
   - Query order_status_transitions for valid to_status_id options
   - Join with order_statuses to get name and code
