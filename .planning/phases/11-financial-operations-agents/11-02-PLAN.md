**5 tool definitions:**

1. `getInventoryStatusTool` (DS-01): name `get_inventory_status`. Queries `products` scoped by `company_id`. Select: id, code, name, stock_quantity, low_stock_threshold, is_active. Input: optional `query` (string, product name/code search), optional `category` (string, category name filter), optional `low_stock_only` (boolean, filter to only items below threshold). Return includes computed `low_stock: stock_quantity <= low_stock_threshold` for each product. Limit results to 20.

2. `getPendingOrdersTool` (DS-02): name `get_pending_orders`. Queries `orders` with order_statuses and order_items. Company-wide scope: `.eq('company_id', context.companyId)` WITHOUT dealer_id filter (warehouse needs all company orders per Pitfall 7 in research). Filter by status codes: 'pending', 'confirmed', 'preparing'. Include dealer info (company_name from dealers join). Input: optional `limit` (number, default 10). First query `order_statuses` to get IDs for the three status codes, then query orders filtered by those status_ids.

3. `updateStockTool` (DS-03): name `update_stock`. WRITE OPERATION. Input: required `product_code` (string), required `new_stock_quantity` (number, must be >= 0). Description MUST include Turkish instruction: "BU ARACI CAGIRMADAN ONCE bayiye guncelleme detaylarini goster ve onay al. Onay alinmadan bu araci ASLA cagirma."
   Handler implementation:
   - Validate product_code is non-empty and new_stock_quantity >= 0
   - Find product by code: `.eq('code', productCode).eq('company_id', context.companyId).eq('is_active', true).single()`
   - If not found, return error
   - Execute UPDATE: `.from('products').update({ stock_quantity: newQuantity, updated_at: new Date().toISOString() }).eq('id', product.id).eq('company_id', context.companyId)` — double company_id scope for defense in depth
   - Return JSON with success, product_code, product_name, previous_quantity, new_quantity
   - Use `(supabase as any)` for the update call (same pattern as satis-tools create_order)

4. `checkReorderLevelTool` (DS-04): name `check_reorder_level`. Queries `products` WHERE `stock_quantity <= low_stock_threshold`. Company-wide scope. Select: code, name, stock_quantity, low_stock_threshold. No required input — returns all below-threshold products. Optional `limit` (default 20).

5. `getShipmentsTool` (DS-05): name `get_shipments`. Queries `orders` with cargo fields (vehicle_plate, driver_name, driver_phone, cargo_notes). Scoped by `dealer_id` for the requesting dealer's shipments (a dealer asks about their own deliveries). Filter: `.not('vehicle_plate', 'is', null)` to only show orders with shipping info. Join `order_statuses` for status name/code. Input: optional `status` (string, filter by order status code like 'shipped'), optional `limit` (number, default 10).
   For status filtering: query `order_statuses` to get the ID for the given code, then add `.eq('status_id', statusRow.id)` to the main query.

**Export `depoSorumlusuTools: Tool[]`** containing all 5 tools.

**Factory function:**
`export function createDepoSorumlusuHandlers(supabase: SupabaseClient<Database>): Map<string, HandlerFn>` — 5 handler entries matching tool names.

Key scope rules from research:
- `get_inventory_status`, `get_pending_orders`, `check_reorder_level`: company_id scope (warehouse is company-wide)
- `get_shipments`: dealer_id scope (dealer sees own shipments)
- `update_stock`: company_id scope on both lookup and update
- `get_pending_orders`: include dealer join for company_name: `.select('order_number, total_amount, created_at, status:order_statuses(name, code), items:order_items(product_name, quantity), dealer:dealers(company_name)')`

Error handling pattern: return `[Hata: ...]` on error, `[Kayit bulunamadi]` on empty results.
