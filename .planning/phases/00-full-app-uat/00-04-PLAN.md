**Gap 1 fix — Report pages (BLOCKER):**

The ExportButton is a `'use client'` component that receives `exportFn` as a prop. The 3 report pages pass inline async closures like `async () => exportSalesReportCSV(period, startDate, endDate)`. In Next.js 16, passing an inline async closure from a server component to a client component without `"use server"` directive causes a runtime serialization error.

Fix approach: Since the export functions in `src/lib/actions/export-reports.ts` are already `'use server'` actions, the problem is the inline wrapper closures that capture local variables (period, startDate, endDate) are NOT server actions themselves. The simplest fix is to use `.bind()` to create bound server actions which Next.js CAN serialize:

In `src/app/(admin)/admin/reports/sales/page.tsx`:
- Replace `exportFn={async () => exportSalesReportCSV(period, startDate, endDate)}` with `exportFn={exportSalesReportCSV.bind(null, period, startDate, endDate)}`
- The `.bind()` pattern on server actions is the Next.js recommended way to pass arguments from server to client components

In `src/app/(admin)/admin/reports/products/page.tsx`:
- Replace `exportFn={async () => exportTopProductsCSV(startDate, endDate)}` with `exportFn={exportTopProductsCSV.bind(null, startDate, endDate)}`

In `src/app/(admin)/admin/reports/dealers/page.tsx`:
- Replace `exportFn={async () => exportDealerPerformanceCSV(startDate, endDate)}` with `exportFn={exportDealerPerformanceCSV.bind(null, startDate, endDate)}`

NOTE: The RPC functions (get_sales_report, get_top_products, get_dealer_performance) are defined in `supabase/migrations/003_reporting_functions.sql`. If they are not yet applied to the DB, the pages will still fail with empty data. This requires a manual SQL execution step on Supabase Dashboard which is covered by Task 3 checkpoint. The code fix above is sufficient for the serialization crash.

**Gap 3 fix — Announcement delete broken:**

The root cause is that RLS silently blocks the UPDATE in `deleteAnnouncement` because `current_company_id()` returns NULL from the admin's JWT. The RLS policy requires `company_id = current_company_id() AND is_company_admin()`. If the JWT hook isn't injecting `company_id`, the policy silently returns 0 rows.

Fix approach: Use the service role client for admin delete operations (bypasses RLS), same pattern as agent layer. This is correct because `deleteAnnouncement` is already an admin-only action.

**Changes to `src/lib/actions/announcements.ts`:**

1. Add import at top: `import { createServiceClient } from '@/lib/supabase/service-client'`

2. **Fix `deleteAnnouncement` function** (lines 325-340):
   - Use service role client instead of session client for the update operation
   - Add row count verification to detect silent failures
   - Keep the soft-delete pattern (is_active = false) which is the established convention
   ```typescript
   export async function deleteAnnouncement(announcementId: string) {
     // Verify caller is authenticated admin first
     const supabase = await createClient()
     const { data: { user } } = await supabase.auth.getUser()
     if (!user) {
       throw new Error('Unauthorized')
     }

     // Check user is admin
     const { data: userData } = await supabase
       .from('users')
       .select('role')
       .eq('id', user.id)
       .single()

     if (!userData || userData.role !== 'admin') {
       throw new Error('Unauthorized: admin only')
     }

     // Use service role client to bypass RLS for admin operation
     const serviceClient = createServiceClient()
     const { data, error } = await serviceClient
       .from('announcements')
       .update({ is_active: false })
       .eq('id', announcementId)
       .select('id')

     if (error) {
       console.error('Error deleting announcement:', error)
       throw new Error('Failed to delete announcement')
     }

     if (!data || data.length === 0) {
       throw new Error('Announcement not found')
     }

     revalidatePath('/admin/announcements')
     revalidatePath('/announcements')
   }
   ```

     // Verify caller is admin
     const { data: { user } } = await supabase.auth.getUser()
     if (!user) throw new Error('Unauthorized')

     const { data: userData } = await supabase

     const content = formData.get('content') as string

5. **Also fix `createAnnouncement` function** with the same service role pattern:
   - Same RLS issue: INSERT requires `company_id` to match `current_company_id()`. Admin user's JWT without company_id claim means INSERT also silently fails.
   - Use service role client, but also need to determine which `company_id` to set. Since this is a single-company setup, query the users table for the admin's company_id.
   ```typescript
   export async function createAnnouncement(formData: FormData) {
     const supabase = await createClient()

     // Verify caller is admin and get company_id
     const { data: { user } } = await supabase.auth.getUser()
     if (!user) throw new Error('Unauthorized')

     const { data: userData } = await supabase
       .from('users')
       .select('role, company_id')
       .eq('id', user.id)
       .single()

     if (!userData || userData.role !== 'admin') throw new Error('Unauthorized: admin only')

     const title = formData.get('title') as string
     const content = formData.get('content') as string
     const priority = parseInt(formData.get('priority') as string) || 0
     const is_active = formData.get('is_active') === 'true'
     const published_at = formData.get('published_at') as string
     const expires_at = formData.get('expires_at') as string

     const serviceClient = createServiceClient()
     const { error } = await serviceClient
       .from('announcements')
       .insert({
         title,
         content,
         priority,
         is_active,
         published_at: published_at || new Date().toISOString(),
         expires_at: expires_at || null,
         ...(userData.company_id ? { company_id: userData.company_id } : {}),
       })

     if (error) {
       console.error('Error creating announcement:', error)
       throw new Error('Failed to create announcement')
     }

     revalidatePath('/admin/announcements')
     revalidatePath('/announcements')
   }
   ```

6. **Fix `getAllAnnouncements`** to also use service role client (admin reads all company announcements):
   ```typescript
   export async function getAllAnnouncements(): Promise<Announcement[]> {
     const supabase = await createClient()

     // Verify caller is admin and get company_id
     const { data: { user } } = await supabase.auth.getUser()
     if (!user) return []

     const { data: userData } = await supabase
       .from('users')
       .select('role, company_id')
       .eq('id', user.id)
       .single()

     if (!userData || userData.role !== 'admin') return []

     const serviceClient = createServiceClient()
     const { data, error } = await serviceClient
       .from('announcements')
       .select('*')
       .eq('company_id', userData.company_id)
       .order('priority', { ascending: false })
       .order('created_at', { ascending: false })

     if (error) {
       console.error('Error fetching all announcements:', error)
       return []
     }

     return data as Announcement[]
   }
   ```

IMPORTANT: The service role client is ONLY used for admin-scoped mutations after verifying the user's admin role via the session-based client. This maintains security: session auth proves identity, service role bypasses broken RLS for the actual DB operation.
