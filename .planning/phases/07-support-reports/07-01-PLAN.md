  <name>Task 1: Create migration with all Phase 7 tables</name>
  <files>supabase/migrations/008_support_reports.sql</files>
  <action>
Create migration file following the exact structure of 007_dashboard_campaigns.sql:

1. **support_messages table**:
   - id (UUID PRIMARY KEY DEFAULT uuid_generate_v4())
   - dealer_id (UUID NOT NULL REFERENCES dealers(id) ON DELETE CASCADE)
   - subject (TEXT NOT NULL)
   - category (TEXT NOT NULL CHECK (category IN ('siparis', 'urun', 'odeme', 'teknik', 'diger')))
   - body (TEXT NOT NULL)
   - status (TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'answered')))
   - reply_body (TEXT) — nullable, set atomically with status change to 'answered'
   - replied_at (TIMESTAMPTZ) — nullable
   - replied_by (UUID REFERENCES auth.users(id)) — nullable, set when admin replies
   - created_at (TIMESTAMPTZ DEFAULT NOW())
   - updated_at (TIMESTAMPTZ DEFAULT NOW())
   - Index on (dealer_id, created_at DESC) for dealer history queries
   - Index on (status, created_at DESC) for admin inbox queries

2. **faq_categories table**:
   - id (UUID PRIMARY KEY DEFAULT uuid_generate_v4())
   - name (TEXT NOT NULL)
   - display_order (INT NOT NULL DEFAULT 0)
   - is_active (BOOLEAN DEFAULT true)
   - created_at (TIMESTAMPTZ DEFAULT NOW())
   - Index on (is_active, display_order)

3. **faq_items table**:
   - id (UUID PRIMARY KEY DEFAULT uuid_generate_v4())
   - category_id (UUID NOT NULL REFERENCES faq_categories(id) ON DELETE CASCADE)
   - question (TEXT NOT NULL)
   - answer (TEXT NOT NULL)
   - display_order (INT NOT NULL DEFAULT 0)
   - is_active (BOOLEAN DEFAULT true)
   - created_at (TIMESTAMPTZ DEFAULT NOW())
   - updated_at (TIMESTAMPTZ DEFAULT NOW())
   - Index on (category_id, display_order)

4. **product_requests table**:
   - id (UUID PRIMARY KEY DEFAULT uuid_generate_v4())
   - dealer_id (UUID NOT NULL REFERENCES dealers(id) ON DELETE CASCADE)
   - product_id (UUID REFERENCES products(id) ON DELETE SET NULL) — nullable: dealer may request a product not yet in catalog
   - product_name (TEXT NOT NULL) — captured at request time even if product deleted later
   - product_code (TEXT) — nullable
   - requested_quantity (INT NOT NULL DEFAULT 1)
   - notes (TEXT) — nullable
   - status (TEXT NOT NULL DEFAULT 'open' CHECK (status IN ('open', 'in_review', 'fulfilled', 'rejected')))
   - admin_notes (TEXT) — nullable, admin response
   - created_at (TIMESTAMPTZ DEFAULT NOW())
   - updated_at (TIMESTAMPTZ DEFAULT NOW())
   - Index on (dealer_id, created_at DESC)

5. **RLS Policies**:

   support_messages:
   - Dealers can SELECT own messages: (SELECT auth.uid()) matches dealer's user_id via dealers JOIN
   - Dealers can INSERT own messages: dealer_id = (SELECT id FROM dealers WHERE user_id = (SELECT auth.uid()))
   - Admin can SELECT all messages: EXISTS (SELECT 1 FROM dealers WHERE user_id = (SELECT auth.uid()) AND role = 'admin') — use existing admin check pattern from financials.ts (get_user_role or service role)
   - Admin can UPDATE messages (for replies): same admin check
   - Use wrapped (SELECT auth.uid()) pattern per Phase 4 RLS decision

   faq_categories and faq_items:
   - All authenticated users can SELECT (no dealer_id filter — FAQ is global)
   - Only service role / admin can INSERT, UPDATE, DELETE

   product_requests:
   - Dealers can SELECT own requests: dealer_id = (SELECT id FROM dealers WHERE user_id = (SELECT auth.uid()))
   - Dealers can INSERT own requests: same pattern
   - Admin can SELECT all: existing admin check pattern
   - Admin can UPDATE requests (status, admin_notes): same admin check

6. **Supabase Realtime publication for support_messages**:
   - GRANT SELECT ON support_messages TO supabase_realtime;
   - ALTER PUBLICATION supabase_realtime ADD TABLE support_messages;
   - Copy exact pattern from 002_realtime_setup.sql (same as orders table setup)
   - This is required for SUP-05: admin receives real-time INSERT events

7. **updated_at triggers**:
   - Add moddatetime trigger on support_messages (updated_at)
   - Add moddatetime trigger on faq_items (updated_at)
   - Add moddatetime trigger on product_requests (updated_at)
   - Use same trigger function as existing tables (check 006 or 007 migration for trigger function name)

8. **Seed data for FAQ** (optional but helpful):
   - Insert 2-3 example faq_categories (e.g., 'Siparis', 'Odeme', 'Urun')
   - Insert 1-2 faq_items per category as examples for admin to replace
