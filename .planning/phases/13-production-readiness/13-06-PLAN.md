**src/lib/__tests__/env.test.ts:**
Test the Zod schema validation (NOT the module-level parse which reads process.env).

Import the schema directly (not the parsed env) and test:
- Valid env object passes validation
- Missing required SUPABASE_SERVICE_ROLE_KEY fails
- Missing required ANTHROPIC_API_KEY fails
- Missing NEXT_PUBLIC_SUPABASE_URL fails
- Invalid NEXT_PUBLIC_SUPABASE_URL (not a URL) fails
- Optional TELEGRAM_BOT_TOKEN_* can be omitted
- Invalid ANTHROPIC_API_KEY (not starting with 'sk-') fails

To avoid triggering the module-level parse, test the schema object itself:
```typescript
// Import only the schema, not the parsed env
import { serverEnvSchema } from '../env'

// Mock all required env vars for the base case
const validEnv = {
  NEXT_PUBLIC_SUPABASE_URL: 'https://test.supabase.co',
  NEXT_PUBLIC_SUPABASE_ANON_KEY: 'test-anon-key',
  SUPABASE_SERVICE_ROLE_KEY: 'test-service-key',
  ANTHROPIC_API_KEY: 'sk-test-key',
  CRON_SECRET: 'test-cron-secret',
}
```

Note: If env.ts runs parse at module level, the test must either:
1. Set process.env before import (using vi.stubEnv), OR
2. Export the schema separately from the parse

The executor should check env.ts implementation and adapt the test accordingly. The key insight: export `serverEnvSchema` separately from the `env` constant so tests can validate the schema without triggering the parse.

**src/lib/__tests__/api-response.test.ts:**
Test the API response helpers:
- `apiSuccess({ id: 1 })` returns NextResponse with `{ success: true, data: { id: 1 }, timestamp: <ISO string> }` and status 200
- `apiSuccess(data, 201)` returns status 201
- `apiError('Bad', 400)` returns `{ success: false, error: { message: 'Bad' }, timestamp }` with status 400
- `apiBadRequest('x')` returns status 400 with code 'BAD_REQUEST'
- `apiUnauthorized()` returns status 401 with default message
- `apiNotFound()` returns status 404
- `apiRateLimited()` returns status 429

**src/lib/__tests__/rate-limit.test.ts:**
Test the rate limiter:
- New RateLimiter with maxRequests=3: first 3 calls to check('key') return allowed=true
- 4th call returns allowed=false with remaining=0
- After window expires (use vi.advanceTimersByTime), calls are allowed again
- cleanup() removes expired entries
- Different keys are tracked independently
- remaining count decreases correctly (3, 2, 1, 0)

Use `vi.useFakeTimers()` and `vi.advanceTimersByTime()` for time-dependent tests.
