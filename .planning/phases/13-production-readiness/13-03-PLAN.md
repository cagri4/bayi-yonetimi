**src/lib/rate-limit.ts:**

Create an in-memory sliding window rate limiter compatible with Vercel Edge runtime (no Node.js-only APIs).

```typescript
type RateLimitConfig = {
  windowMs: number    // Time window in milliseconds
  maxRequests: number // Max requests per window
}

class RateLimiter {
  private store: Map<string, { count: number; resetAt: number }>

  constructor(private config: RateLimitConfig)

  check(key: string): { allowed: boolean; remaining: number; resetAt: number }

  // Periodic cleanup of expired entries (call in middleware)
  cleanup(): void
}
```

Implementation:
- Use Map<string, { count: number, resetAt: number }> for storage
- `check(key)`: If no entry or expired, create new window. If within window, increment count. Return allowed (count <= max), remaining, resetAt.
- `cleanup()`: Delete entries where Date.now() > resetAt. Call every 60 seconds (track lastCleanup timestamp).
- Store is module-level (survives across requests within same serverless instance, resets on cold start).

Export pre-configured instances:
- `apiLimiter` = new RateLimiter({ windowMs: 60_000, maxRequests: 60 }) — 60 req/min for general API
- `telegramLimiter` = new RateLimiter({ windowMs: 60_000, maxRequests: 30 }) — 30 req/min per chat for Telegram webhooks
- `cronLimiter` = new RateLimiter({ windowMs: 60_000, maxRequests: 5 }) — 5 req/min for cron endpoints

**src/lib/logger.ts:**

Create a structured logger that outputs JSON to console (Vercel captures console output for log drains).

```typescript
type LogLevel = 'debug' | 'info' | 'warn' | 'error'

type LogEntry = {
  level: LogLevel
  message: string
  requestId?: string
  timestamp: string
  [key: string]: unknown
}

function createLogger(context?: { requestId?: string; module?: string }): Logger

interface Logger {
  debug(message: string, meta?: Record<string, unknown>): void
  info(message: string, meta?: Record<string, unknown>): void
  warn(message: string, meta?: Record<string, unknown>): void
  error(message: string, meta?: Record<string, unknown>): void
}
```

Implementation:
- Each log method creates a LogEntry with level, message, timestamp (ISO), requestId (from context), module (from context), and spreads meta.
- Output via `console.log(JSON.stringify(entry))` for info/debug, `console.warn(JSON.stringify(entry))` for warn, `console.error(JSON.stringify(entry))` for error.
- Export a default `logger` instance with no requestId (for module-level logging).
- Export `createLogger` factory for request-scoped loggers.

Note: Do NOT import in Edge middleware directly — logger is for API route and server-side use. Middleware uses simple console.log for now (Edge runtime compatibility).

Update src/middleware.ts to add request ID generation and rate limiting.

**Request ID (all requests):**
- At the very start of the middleware function, generate `requestId = crypto.randomUUID()` (available in Edge runtime).
- Set `x-request-id` header on the response (both supabaseResponse and any redirect responses).

**Rate limiting (API routes only):**
- Import rate limiter. NOTE: Since middleware runs on Edge and rate-limit.ts uses Map, it IS compatible (Map is available in Edge runtime).
- After the existing publicPaths check, add rate limiting logic for /api/ routes:
  1. Extract client IP from `request.headers.get('x-forwarded-for')?.split(',')[0]?.trim() || request.ip || 'unknown'`
  2. For `/api/telegram/*` routes: use `telegramLimiter.check(ip)`. Also run `telegramLimiter.cleanup()`.
  3. For `/api/cron/*` routes: use `cronLimiter.check(ip)`. Also run `cronLimiter.cleanup()`.
  4. For other `/api/*` routes: use `apiLimiter.check(ip)`. Also run `apiLimiter.cleanup()`.
  5. If `!result.allowed`: return `NextResponse.json({ success: false, error: { message: 'Too many requests', code: 'RATE_LIMITED' } }, { status: 429, headers: { 'x-request-id': requestId, 'Retry-After': String(Math.ceil((result.resetAt - Date.now()) / 1000)), 'X-RateLimit-Remaining': '0' } })`
  6. If allowed: add `X-RateLimit-Remaining` header to supabaseResponse.

**Integration with existing middleware:**
- Keep ALL existing logic (auth, redirects, admin protection) unchanged.
- Rate limiting runs BEFORE auth checks for /api/ routes (return early on 429).
- Request ID is added to ALL responses (including auth redirects).

When creating redirect responses (e.g., `NextResponse.redirect(loginUrl)`), set the `x-request-id` header on the redirect response as well.

For supabaseResponse, set headers via `supabaseResponse.headers.set('x-request-id', requestId)` before returning.
